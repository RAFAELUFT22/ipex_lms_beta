Estratégia para o Self-Hosted (Correção de Erros)

O erro "400 Bad Request" e "Network Error" no provisionamento provavelmente ocorreu porque o Supabase é uma stack pesada (10+ containers) e o Dokploy falhou ao tentar processar o arquivo docker-compose via API, ou houve conflito de portas.

A

Deploy Nativo (Recomendado)

Rodar o Supabase diretamente via terminal (docker compose up -d) na máquina, contornando o Dokploy.

Evita o erro 400 da API do Dokploy e garante que os containers subam corretamente. Vamos conectá-lo à rede do n8n/LMS.
B

Insistir no Dokploy

Ajustar o docker-compose do Supabase, remover conflitos de porta e tentar criar manualmente pelo painel do Dokploy.

Mantém tudo no painel do Dokploy, mas corremos o risco de esbarrar novamente nos limites da API.

Principais Ajustes Necessários (Para ambas as opções):

  • Portas: O Supabase usa a porta 8000 por padrão, que já está sendo usada pelo Frappe Backend. Precisamos alterar para 8001 ou similar.
  • Banco de Dados: O Supabase exige seu próprio Postgres. Ele vai rodar em paralelo com o seu Postgres atual (do Evolution/n8n/Chatwoot).
  • Rede: Precisamos colocar o Supabase na rede kreativ_net para que o n8n e a nova API do LMS consigam conversar com ele.
